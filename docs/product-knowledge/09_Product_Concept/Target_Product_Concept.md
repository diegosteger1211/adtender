# Target Product Concept — adtender Design Vision

> This document consolidates all reference platform observations and product knowledge into adtender's unified design vision. Reference platforms informed this document; they did not define it. adtender is its own product. Every concept below is an adtender-native decision.

---

## 1. Overall Product Philosophy

### 1.1 What Makes an Enterprise Tender Platform Successful

Enterprise procurement teams operate under three simultaneous pressures:

1. **Compliance pressure** — Every decision must be defensible. Audit trails, equal treatment, and process integrity are non-negotiable.
2. **Speed pressure** — Tender cycles are long by nature. Every unnecessary click or tool switch makes them longer.
3. **Coordination pressure** — Procurement involves project managers, legal reviewers, evaluators, decision board members, and suppliers. Everyone needs the right information at the right time without stepping on each other.

A successful enterprise tender platform resolves this tension rather than collapsing under it. It does this by:

- **Making the process visible** — Users always know where they are in the lifecycle and what comes next. The platform is a guide, not a form.
- **Enforcing compliance invisibly** — Business rules are built into the flow. Users do not need to remember them; the platform prevents violations before they happen.
- **Centering on work, not navigation** — The user's current task is always in focus. Getting to the next action takes one click, not three.
- **Preserving every decision** — Nothing is lost. Every action, score, answer, and approval is recorded. The record is the product.

### 1.2 Concepts adtender Adopts

| Concept | Origin | Why adtender adopts it |
|---|---|---|
| **Two-level navigation** (module + lifecycle stage) | Enterprise procurement platforms | The two mental modes (where am I / what phase am I in) map directly to how procurement professionals think |
| **Requirement × Supplier matrix** | Evaluation tooling | The most natural analytical frame for multi-supplier, multi-requirement evaluation |
| **Threaded clarification Q&A** | Enterprise communication tools | Structured threads are auditable; unstructured email is not |
| **Stage-aware action visibility** | Workflow platforms | Prevents out-of-sequence operations without error messages |
| **System-generated tasks** | Workflow automation | Reduces manual coordination overhead and missed deadlines |
| **Role-based dashboard** | Analytics platforms | Different roles need different information first |

### 1.3 Concepts adtender Improves

| Observed Weakness | adtender Improvement |
|---|---|
| Stage navigation is passive (informational only) | Stage bar is active: completed stages link to phase summaries; clicking the active stage navigates to the primary workspace |
| Clarification buried in a tender tab | Clarification is a first-class workspace with its own navigation entry, draft/review/publish workflow |
| Supplier offer view requires cross-screen navigation | Consolidated Offer Detail panel (5 tabs) provides everything about one supplier in one place |
| Evaluation matrix cells are static indicators | Cells are interactive: click for detail, drill to evaluator scores, surface AI variance signals |
| Export creates static unversioned files | All exports are version-stamped — every export references the exact Business Object versions for audit traceability |
| AI absent | AI assists in every workspace — scoring, drafting, detecting anomalies — but never makes binding decisions |
| Tasks are manually created | Tasks are generated by domain events; manual tasks supplement rather than replace |
| Configuration changes are silent | All configuration changes produce audit events; changes affecting active tenders show an impact warning |

## 2. Navigation Concept

### 2.1 Design Principle

Navigation in adtender operates on two independent axes:

- **Horizontal axis (lifecycle)**: Where is this tender in its process? — communicated by the stage progress bar
- **Vertical axis (module)**: Which functional area am I working in? — communicated by the left sidebar

These two axes are always visible simultaneously in any tender context. Outside a tender, only the sidebar is active.

### 2.2 Main Navigation (Left Sidebar)

The sidebar is the permanent orientation element. It is always visible on desktop (240px expanded / 56px collapsed).

**Platform-level modules** (always available):

| Item | Icon | URL |
|---|---|---|
| Dashboard | ⊞ | `/` |
| Projects | 📁 | `/projects` |
| Requirement Library | 📚 | `/library` |
| Suppliers | 🏢 | `/suppliers` |
| Tasks | ✓ | `/tasks` |
| Reports | 📊 | `/reports` |
| Administration | ⚙ | `/admin` |

**Project context section** (when inside a project):

```
▸ PROJECT: [Project Name]
  Overview
  Requirements
  Tenders
  Team
```

**Tender context section** (when inside a tender):

```
▸ TENDER: [Tender Name]
  Overview
  Requirements
  Suppliers
  Clarifications         [lock indicator if unavailable]
  Evaluation             [lock indicator if unavailable]
  Decision               [lock indicator if unavailable]
```

Lock indicators show a `🔒` icon with a tooltip explaining the prerequisite. Clicking a locked item is non-destructive — it displays the tooltip without navigating.

### 2.3 Stage Progress Bar (Tender Context)

The stage bar appears on every tender workspace screen, below the workspace header and above the tab navigation. It is 44px tall and spans the full content width.

```
✅ Setup ──── ✅ Published ──── ✅ Responses ──── 🔄 Evaluation ──── ⬜ Decision ──── ⬜ Awarded
   Jun 1           Jun 1            Jun 15             Jul 22 →             —               —
```

Stage states:

| State | Visual | Click behavior |
|---|---|---|
| Completed | ✅ green + date | Opens read-only phase summary panel |
| Active | 🔄 blue highlight | Navigates to the primary tab for this stage |
| Upcoming | ⬜ gray | Tooltip: what prerequisite must be met |
| Overdue | 🔴 red | Tooltip: phase has passed expected date |

The stage bar drives action visibility: the primary CTA in the workspace header changes with each stage (see §4 Functional Modules).

### 2.4 Context Navigation (Tabs)

Within any workspace, a tab row provides access to sub-views. Tabs are context-sensitive: which tabs are visible and which are locked depends on the lifecycle stage.

Tabs are never hidden — locked tabs are shown in a muted state with a tooltip. This lets users understand what is coming without being confused about where things went.

### 2.5 Breadcrumbs

The top bar breadcrumb tracks the user's position in the hierarchy:

```
Projects › IT Modernization › IT Security Suite › Evaluation
```

In tender context, the last segment reflects the current stage, not just the module name. Each segment is a clickable link.

### 2.6 Favorites

Users can bookmark any record (project, tender, supplier, requirement) via a star icon `☆ → ★`. Favorites appear:
- As a pinned section at the top of the relevant list view
- In the Global Search recent items panel
- In the sidebar under a collapsible "Favorites" section (max 8 items)

Favorites are personal, per-user, and persisted across sessions.

### 2.7 Global Search

Activated by `Cmd/Ctrl + K` from anywhere in the application.

**Default state** (no input): Recent items + Quick Actions panel.

**Typing behavior**: Results appear after 2 characters, grouped by type:

| Group | Examples |
|---|---|
| Projects | IT Modernization 2024 |
| Requirements | REQ-0042 — Data Encryption at Rest |
| Tenders | TND-2024-0007 — IT Security Suite |
| Suppliers | Acme Systems AG |
| Library | Requirement Library — Security Standards |
| Decisions | Decision 2024-003 — IT Security Suite |

Ranking order: exact code match → title prefix → full-text match → recency.

All results are tenant-scoped and permission-scoped. Users never see records outside their tenant or their access level.

**Command palette**: typing `/` switches to command mode. Available commands:

```
/new-project    Create a new project
/new-tender     Start the tender wizard
/new-requirement  Create a new requirement
/new-supplier   Register a new supplier
```

**Keyboard navigation**: `↑↓` to navigate, `Enter` to select, `→` to preview in right panel, `Esc` to close.

### 2.8 Notifications

Notifications are delivered via three channels: in-app panel, email, and toast messages.

**In-app panel** (bell icon, top bar):
- Tabs: All / Unread / Action Required
- Four types with color coding:
  - 🔴 Action Required — COI declarations, evaluation deadlines, review assignments
  - 🔵 Informational — approvals, status changes, submissions received
  - 🟣 AI Insights — always dismissible; never mandatory
  - 🟠 System Alerts — storage warnings, expired invitations

**Toast messages**: 4-second timeout; Undo available for reversible actions (e.g., bulk delete, status change).

**Email notifications**: Split into non-configurable (COI declarations, decision approvals, submission received) and configurable (summary digests, reminders, AI insights).

All notifications are driven by domain events. No notification is generated without a corresponding audit record.

## 3. Workspace Concept

### 3.1 Standard Layout

All workspaces share a single layout anatomy. Consistency is the primary usability multiplier: a user who learns one workspace understands all of them.

```
┌─────────────────────────────────────────────────────────────────────────┐
│  TOP BAR: Breadcrumb · Global Search · Notifications · User menu        │
├──────────────┬──────────────────────────────────────────────────────────┤
│              │  [STAGE PROGRESS BAR — tender context only]              │
│  LEFT        ├──────────────────────────────────────────────────────────┤
│  SIDEBAR     │  WORKSPACE HEADER                                        │
│  (240px)     │  Title · Record Code · Status badge · Primary CTA        │
│              ├──────────────────────────────────────────────────────────┤
│  Module      │  TOOLBAR                                                 │
│  navigation  │  [Search] [Filters] [View toggle] [Bulk actions] [Export]│
│              ├──────────────────────────────────────────────────────────┤
│              │  FILTER CHIPS (active filters)                           │
│              ├──────────────────────────────────────────────────────────┤
│              │  MAIN TABLE / CONTENT          │  DETAIL PANEL (right)   │
│              │                               │  (opens on row select)   │
│              │                               │                         │
│              │                               │  [Detail tabs]          │
│              │                               │  [Context actions]      │
│              │                               │  [AI panel]             │
│              │                               │  [Timeline]             │
│              │                               │  [Comments]             │
│              │                               │  [Attachments]          │
└──────────────┴───────────────────────────────┴─────────────────────────┘
```

### 3.2 Toolbar

The toolbar sits directly below the workspace header (or below the stage bar in tender context).

Standard toolbar zone layout:

```
LEFT ZONE                              RIGHT ZONE
[🔍 Search...] [Filter ▾] [Group ▾]   [View: Table ▾] [Bulk ▾] [Export ▾] [+ New]
```

- `Search`: always instant (filter-as-you-type, client-side on loaded records; server-side for full-text)
- `Filter`: opens filter panel below toolbar; active filters shown as removable chips
- `Group`: groups rows by a selected column (category, status, assignee)
- `View toggle`: switches between Table / Card / Kanban / Calendar where applicable
- `Bulk`: available when rows are selected; contains bulk operations for that workspace
- `Export`: contextual export options for the current view
- `+ New`: primary creation action for the workspace

### 3.3 Filter Area

Filters appear as chips directly below the toolbar when active:

```
Active filters:  [Status: Open ✕]  [Supplier: Acme ✕]  [Priority: Critical ✕]  [Clear all]
```

Each chip is removable. "Clear all" removes all active filters. Filter state is persisted per user per workspace.

**Filter panel** (opens below the toolbar):
- Multi-select options per filter dimension
- Range filters for dates and numeric values
- Saved filters: users can save a filter set as "My Filter — Active Criticals" and load it from a dropdown

### 3.4 Main Table

The main table is the primary data view. Standard columns:

- Checkbox (bulk selection)
- Record code (e.g., REQ-0042, TND-2024-007)
- Title / Name (primary content column, truncated with tooltip)
- Status badge
- Key date
- Owner / Assignee (avatar + name)
- Last modified
- Actions column (⋮ hover menu)

**Column behavior:**
- Column widths are resizable via drag on the header border
- Columns are reorderable via drag
- Column visibility is configurable per user (save to "my columns")
- Clicking a row selects it and opens the Detail Panel; double-click opens full-page detail
- Rows are selectable for bulk operations
- Sorting: click column header; click again to reverse; Shift+click for multi-sort

**Virtual scrolling**: tables with 100+ rows use virtual scrolling. No pagination.

### 3.5 Detail Panel

The Detail Panel opens on the right when a row is selected. It is a 40% width panel that does not push the table (overlay on the right).

The Detail Panel has tabs for each data category relevant to the record type:

| Workspace | Detail Panel tabs |
|---|---|
| Requirements | Details / History / Related / AI / Comments |
| Tenders | Overview / Requirements / Suppliers / Evaluation summary |
| Suppliers | Overview / Qualifications / Certifications / Tenders / History |
| Clarifications | Thread / Related Requirement / Audit |
| Evaluations (cell) | Evaluators / Supplier Response / AI insight |

**Inline editing in the Detail Panel:** Fields that can be edited are shown with a pencil icon on hover. Clicking activates an inline editor. Changes auto-save; no separate Save button for field-level edits.

Closing the Detail Panel returns focus to the row in the table.

### 3.6 Context Actions

Context actions are operations that apply to the currently selected record. They appear in two places:

1. **In the Detail Panel header**: Primary action button + secondary action dropdown `[Manage ▾]`
2. **In the table row hover menu** (⋮): Quick actions without opening the Detail Panel

Context actions are lifecycle-aware: only valid actions for the current state are shown. Invalid actions are hidden, not disabled (to avoid confusion).

### 3.7 AI Assistant Panel

The AI panel is a collapsible section at the bottom of the Detail Panel or as a dedicated tab. It is not a floating chat assistant — it is contextually scoped to the record being viewed.

**AI panel structure:**

```
✦ AI INSIGHTS                                        [Collapse]
─────────────────────────────────────────────────────────────
✦  [Insight text relevant to this specific record]
   [Action link]                          [Accept ✓] [Dismiss ✕]

✦  [Second insight]
   [Accept ✓] [Edit ✎] [Dismiss ✕]
─────────────────────────────────────────────────────────────
[Ask AI about this record...]
```

**AI interaction rules:**
- Every AI output is marked with the `[✦ AI]` chip
- Three response options: Accept (applies the suggestion), Edit (opens suggestion in editor), Dismiss (removes it)
- AI never auto-applies any change
- AI insights are per-record, not global
- Users can ask a free-text question about the current record via the input at the bottom
- AI panel state (expanded/collapsed) is persisted per workspace per user

### 3.8 Timeline

The Timeline is a chronological activity log for the current record. It appears as a tab in the Detail Panel or as a dedicated History tab in full-page views.

Each entry shows:
- Date and time
- Actor (user or System)
- Event description (plain language, not technical event names)
- Changed fields (old value → new value) for edit events
- Link to the related document or record if applicable

The Timeline is immutable. No entry can be deleted or edited. It is the primary audit artifact.

### 3.9 Comments

The Comments section is attached to records that support collaborative annotation: Requirements, Clarification threads, Decisions.

Comments are not chat — they are contextual annotations on a specific record. Each comment:
- Belongs to a specific record
- Can mention a user (`@name`)
- Can reference another record (`#REQ-0042`)
- Is permanently stored and included in the Timeline

Comments are visible to all users with read access to the record. They are not private.

### 3.10 Attachments

The Attachments section is available on records that support file evidence: Requirements, Supplier Responses, Decisions, Clarifications.

Standard behavior:
- Drag-and-drop to upload
- File preview for PDF, images, and Office documents (where supported)
- Version: uploading a new file with the same name creates a new version (both preserved)
- Each attachment records: uploader, upload date, file size, linked record
- Download: individual file or all as ZIP
- Attachments are included in export packages

## 4. Functional Modules

### 4.1 Dashboard

**Purpose:** Orientation and action hub. The user's first screen. Every element is a navigation entry point.

**Key panels:**
- **Priority Tasks** (pinned top): up to 5 overdue/today tasks with direct action links
- **Active Tenders**: card per tender with mini stage bar, deadline countdown, progress indicator, one alert
- **Upcoming Deadlines**: compact 14-day timeline across all tenders
- **My Projects**: compact project cards with health indicator
- **AI Insights**: up to 3 advisory items, always dismissible

**Role variants:** Procurement Manager, Project Manager, Evaluator, Decision Board Member, Org Admin. Each role sees the panels most relevant to their responsibilities in priority order.

### 4.2 Projects

**Purpose:** Top-level container for procurement initiatives. Groups tenders, team, and milestones.

**Key capabilities:**
- Create / archive project
- Project overview with stage bar (Initiated → Active → TenderRunning → Closing → Archived)
- Tender list within project
- Team member management
- Lessons Learned capture at project close
- Requirements scoped to project (project-specific, not library)

### 4.3 Requirement Libraries

**Purpose:** Reusable requirement repository. The source from which tenders draw their requirement sets.

**Key capabilities:**
- Library catalog (browse / filter / search)
- Create and manage multiple libraries (e.g., by domain: Security, Data Privacy, Integration)
- Library-level approval workflow
- Import library requirements into tender
- Archiving a library preserves all requirements (does not deprecate them)
- Knowledge Flywheel: Lessons Learned feeds new or revised requirements back to the library

### 4.4 Requirements

**Purpose:** Individual requirement authoring and lifecycle management.

**Key capabilities:**
- Create / edit requirement with quality score (AI-computed, real-time)
- Lifecycle: Draft → Review → Approved → Deprecated
- Versioning: every Approval creates a new immutable version
- Import from library into project scope
- Category / priority / weight assignment
- AI: duplicate detection, wording improvement, category suggestion
- Kanban view: drag cards between lifecycle stages
- Bulk status operations

### 4.5 Tender Wizard

**Purpose:** Guided 5-step creation of a publication-ready tender.

**Key capabilities:**
- Step 1: Basics (title, type, description, template)
- Step 2: Requirements (select from library; set evaluation weights)
- Step 3: Suppliers (invite Qualified SupplierProfiles)
- Step 4: Settings (deadlines, clarification config, anonymization toggle)
- Step 5: Review (checklist; publish or save draft)
- AI: template recommendation, requirement gap detection, supplier suggestions
- Publication: freezes requirement versions; notifies all invited suppliers immediately

### 4.6 Tender Management

**Purpose:** Full lifecycle management of a published tender.

**Key capabilities:**
- Stage progress bar driving action visibility
- Overview: progress summary, timeline, next-phase checklist
- Requirements tab: frozen snapshot after publication; version-pinned
- Suppliers tab: response status, portal access, resend/revoke
- Clarifications tab: pointer to Clarification Workspace
- Evaluation tab: pointer to Evaluation Workspace
- Decision tab: pointer to Decision Workspace
- Lifecycle CTAs: Publish → Close Submissions → Start Evaluation → Lock Evaluations → Open Decision Board → Award
- Export: tender document PDF, requirement snapshot CSV, audit trail PDF

### 4.7 Supplier Management

**Purpose:** Organizational supplier registry. Pre-tender qualification. Portal access control.

**Key capabilities:**
- Supplier registry: search, filter by qualification status, category, portal access
- Register new supplier (with AI duplicate detection)
- Qualification workflow: 3-step guided panel (documents → review → approve)
- SupplierProfile detail: 5 tabs (Overview / Qualifications / Certifications / Tenders / History)
- Certification expiry tracking with color coding (green / amber / red)
- Portal access management: activate, revoke, resend invitation

### 4.8 Supplier Portal

**Purpose:** Supplier-facing application. Isolated from the buyer application. Suppliers see only their own data.

**Key capabilities:**
- Tender invitation dashboard with deadline countdown
- Tender response: per-requirement compliance + description + evidence upload
- Auto-save every 30 seconds
- Submit response with confirmation dialog
- View all published clarification answers (anonymized)
- Post clarification questions (before deadline)
- Profile management (contact details)
- Read-only view after submission

### 4.9 Clarifications

**Purpose:** Structured Q&A management. Enforces equal treatment. Produces formal audit record.

**Key capabilities:**
- Thread list: Open / Draft / Answered / Published to All
- Thread detail: question + related requirement + answer editor + publish options
- Publish to All (default): sends anonymized answer to all invited suppliers
- Private answer: sends only to the asking supplier
- Amendment notice: formal change record with optional deadline extension
- Clarification Log export (PDF)
- AI: answer drafting, duplicate detection, equal treatment flag, amendment impact analysis
- Post-deadline: buyer can still answer; supplier questions blocked

### 4.10 Evaluation Workspace

**Purpose:** Structured scoring of supplier responses. Enforces blind scoring until lock.

**Key capabilities (pre-lock):**
- Assign evaluators to suppliers (and requirements)
- Monitor completeness via Evaluation Matrix (M×N grid)
- Reminder sending to late evaluators
- Individual scoring interface: 1–10 buttons, rationale for extreme scores, AI suggestion chip per requirement
- Blind scoring: no other evaluator's scores visible until all locked

**Key capabilities (post-lock):**
- Score heatmap revealed
- Side-by-side comparison
- Variance detection (⚡ > 2.0 std dev)
- Calibration session support
- Export: matrix XLSX/PDF, gap report, calibration report

### 4.11 Decision Workspace

**Purpose:** Formal procurement decision. Board management, COI, approval, immutable record.

**Key capabilities:**
- Configure decision board (members + chair)
- COI declaration per supplier per board member
- Redacted report access for conflicted members
- Record Decision: awarded supplier, justification, conditions, trade-offs
- Board approval: all members must approve; checkboxes per member
- Immutable once approved (GBR-017)
- Revocation: creates new Decision aggregate; original preserved
- Discussion thread for board members
- History: COI events, report access log, discussion

### 4.12 Reports

**Purpose:** Cross-workspace reporting and export for procurement documentation and internal governance.

**Key report types:**
- Tender Summary Report (per tender, all phases)
- Evaluation Matrix Report (full scored comparison)
- Clarification Log (Q&A audit record)
- Decision Report (award + justification + approvals)
- Audit Trail Report (full event history for a tender)
- Deadline Summary (team planning view)
- Supplier Participation Report (across multiple tenders)

All reports are version-stamped with the Business Object version IDs at the time of generation.

### 4.13 Administration

**Purpose:** Tenant-level configuration and user management.

**Key capabilities:**
- User management: invite, suspend, deactivate, manage roles
- Role assignment (Project Manager, Procurement Manager, Evaluator, Board Member, Admin)
- Organization profile (legal name, contact, logo)
- Requirement category configuration
- Tender type configuration
- Supplier category configuration
- Audit log viewer (read-only)
- System alerts management

## 5. UX Principles

### 5.1 Minimal Clicks

Every primary workflow should be reachable in 3 clicks or fewer from the dashboard. Measured as:
- Dashboard → Tender → Primary action: 2 clicks
- Dashboard → Task action link: 1 click
- Global Search → Record: 2 keystrokes + Enter

Navigation depth beyond 3 levels signals a design problem, not a complexity requirement.

### 5.2 No Unnecessary Dialogs

A confirmation dialog must only appear when the action is irreversible or has significant consequences. It must explain the consequence in plain language.

Dialogs are NOT appropriate for:
- Saving a draft
- Adding a record to a list
- Changing a filter

Dialogs ARE appropriate for:
- Publishing a tender (notifies all suppliers immediately)
- Locking evaluations (irreversible)
- Approving a decision (immutable record)

### 5.3 Inline Editing

Field-level editing happens in context. No separate Edit page for routine field changes. Click a field → it becomes editable → blur or Enter saves. Changes auto-save with a brief confirmation toast.

Inline editing is only available for fields that are safe to change at that lifecycle stage. Frozen fields (e.g., requirements after publication) show a lock icon with an explanatory tooltip.

### 5.4 Drag and Drop

Drag-and-drop is available where reordering or reclassification is a natural gesture:
- Requirement Kanban: drag cards between lifecycle stages
- Tender Wizard Step 2: drag requirements from library to selected list
- Requirement evaluation weight rows: drag to reorder
- Dashboard panels: drag to reorder (non-pinned panels only)

Drag-and-drop always has a keyboard-accessible alternative.

### 5.5 Reusable Workspaces

A workspace is a persistent, stateful environment, not an isolated page. State that persists per user per workspace:
- Active filters
- Selected view mode (table / card / kanban)
- Column configuration
- Saved filter sets
- Detail Panel open/closed state
- Last selected record

Users return to their workspace as they left it. A fresh browser session restores their context.

### 5.6 Configurable Views

Every list view supports at minimum: Table view. Where workflow benefits from it: Card view, Kanban view, Calendar view. Users choose their preferred view per workspace. The choice persists.

Column configuration (add/remove/reorder columns) is available in all table views. Users can save a column configuration as a named view ("My Evaluation View").

### 5.7 Saved Filters

Users can save any active filter combination as a named filter:
- Name (free text)
- Scope (personal only)
- Applied from a dropdown in the filter toolbar

Saved filters appear in the filter dropdown for that workspace. Deleting or renaming is available. Filter sets do not have a limit.

### 5.8 Keyboard Shortcuts

Every workspace has a published keyboard shortcut set. Shortcuts follow a consistent mnemonic pattern:

| Shortcut | Action |
|---|---|
| `Cmd/Ctrl + K` | Open Global Search |
| `N` | New record (workspace-specific) |
| `E` | Edit selected record |
| `F` | Focus filter input |
| `Esc` | Close Detail Panel / dialog |
| `Tab / Shift+Tab` | Navigate wizard steps |
| `↑↓` | Navigate rows in table |
| `Enter` | Open selected row in Detail Panel |
| `1–9 / 0` | Score buttons (Evaluation Workspace only) |
| `A / D` | Accept / Dismiss AI suggestion |

`?` opens a keyboard shortcut reference overlay in any workspace.

### 5.9 Responsive Design

| Breakpoint | Behavior |
|---|---|
| ≥ 1440px | Full layout; sidebar 240px; detail panel 40% |
| 1024–1439px | Sidebar collapses to 56px icons; detail panel 380px |
| 768–1023px | Sidebar collapses to drawer (hamburger); stage bar condensed; detail panel full-width overlay |
| < 768px | Mobile layout; sidebar as bottom sheet; stage bar as single chip; no detail panel (full-page navigation) |

Core workflows (dashboard, task actions, supplier portal) are fully functional on tablet. The Supplier Portal is optimized for mobile (suppliers may be on any device).

### 5.10 Auditability

Every binding action in adtender produces an immutable audit record. This is not a logging feature — it is a product commitment.

Users experience auditability as:
- Timeline visible on every record
- Export of full Audit Trail available on any tender
- Notification when a record they own is changed by another user
- Version identifiers visible on hover for any versioned field

No record is deleted. Records are Archived (soft-closed, read-only). Users cannot permanently remove data.

### 5.11 Enterprise Usability

Enterprise usability means the product works for experienced users who do it every day, not just first-time users exploring features.

Design priorities for experienced users:
- Keyboard-first interactions are always available
- Bulk operations for repeated actions (select 12 rows, reassign all)
- Saved views and filters eliminate repetitive configuration
- Deep-link every record (share a URL to any specific tender, requirement, evaluation cell)
- Power features accessible without being in the way of basic use

The platform does not optimize for onboarding at the expense of daily efficiency.

## 6. AI Integration

### 6.1 AI Philosophy

AI in adtender is an **assistant**, not an actor. It assists users in doing their work better. It does not make business decisions.

The boundary is absolute: AI can draft an answer, compute a quality score, flag an anomaly, and suggest a supplier. The procurement professional reads, judges, and decides.

### 6.2 AI Output Pattern

Every AI output follows the same pattern:

```
✦ [AI Insight or suggestion text]
[Accept ✓]  [Edit ✎]  [Dismiss ✕]
```

- **Accept**: applies the suggestion without opening an editor
- **Edit**: opens the suggestion in an editable field (user can modify before applying)
- **Dismiss**: removes the suggestion (does not apply it)

AI never auto-applies. The `[✦ AI]` chip is the visual identifier across all workspaces.

### 6.3 AI Capabilities by Workspace

**Dashboard**
- At-risk task identification: flags tasks likely to miss deadline based on activity patterns
- Workload imbalance: alerts if one team member has a disproportionate task load
- Clarification deadline warning: surfaces open questions near deadline

**Requirement Library & Editor**
- Quality score (1–10): computed in real-time as author types; based on clarity, specificity, completeness
- Duplicate detection: warns when a similar requirement exists in the library
- Wording improvement: suggests clearer phrasing for ambiguous requirements
- Category suggestion: proposes a category based on requirement content
- Reuse suggestion: "This requirement already exists in Library X — consider importing"

**Tender Wizard**
- Template recommendation: suggests a starting template based on tender type and category
- Requirement gap detection: "No Data Privacy requirements detected — relevant for this category"
- Supplier suggestion: recommends qualified suppliers from registry based on tender category and past performance

**Clarification Workspace**
- Answer drafting: generates a proposed answer from the requirement text and acceptance criteria
- Duplicate question detection: "Similar question Q-2024-0012 exists — consider consolidating"
- Equal treatment flag: "This answer appears material to all suppliers — consider publishing to all"
- Amendment impact analysis: "This answer may change supplier response requirements for REQ-0042"

**Evaluation Workspace**
- Per-requirement suggestion: AI scoring hint based on supplier response content (evaluator makes the final score)
- Variance alert: ⚡ flags cells where evaluator scores diverge > 2.0 standard deviations
- Calibration suggestion: "Evaluator X's scores are consistently 1.8 points lower — possible calibration bias"
- Differentiating requirements: identifies which requirements show the highest score spread
- Completion risk: flags at-risk cells with <24h to deadline

**Decision Workspace**
- Decision justification draft: generates a draft justification from evaluation data
- Comparative summary: natural language narrative comparing top 2–3 suppliers across key dimensions
- Risk flag: highlights supplier-specific risk factors surfaced during evaluation

**Reports**
- Executive summary generation: produces a one-page summary of a completed tender
- Gap narrative: explains which suppliers scored below threshold and why

### 6.4 AI Boundaries

AI shall never:
- Approve a requirement, tender, evaluation, or decision
- Publish an answer to a supplier
- Assign a score on behalf of an evaluator
- Issue an amendment notice
- Award a tender to a supplier
- Access data outside the user's tenant

AI output is always visually distinguished from human-authored content by the `[✦ AI]` chip. Users can always identify what was written by AI.

## 7. MVP Prioritization

### 7.1 Phase 1 — Foundation

**Goal:** Users can manage projects, build a requirement library, author requirements.

| Module | Key Deliverables |
|---|---|
| Dashboard | Role-based dashboard, Priority Tasks panel, Active Tenders panel |
| Projects | Create / manage / archive project; project lifecycle stage bar; team management |
| Requirement Library | Library catalog; create/edit/approve requirements; versioning; AI quality score |
| Requirement Editor | Full lifecycle (Draft → Review → Approved); inline editing; kanban; import from library |

**Exit criterion:** A procurement manager can create a project, build a library, and author an approved set of requirements.

### 7.2 Phase 2 — Tender Execution

**Goal:** Users can create and publish a tender; suppliers can respond.

| Module | Key Deliverables |
|---|---|
| Tender Wizard | 5-step wizard; publication; requirement snapshot freeze; supplier notification |
| Tender Workspace | Stage bar; lifecycle CTAs; requirements tab; suppliers tab; response monitoring |
| Supplier Management | Registry; qualification workflow; portal access management |
| Supplier Portal | Invitation dashboard; response editor; auto-save; submission |

**Exit criterion:** A procurement manager can publish a tender and receive submitted responses from suppliers.

### 7.3 Phase 3 — Evaluation and Decision

**Goal:** Users can evaluate responses and record a formal decision.

| Module | Key Deliverables |
|---|---|
| Clarifications | Q&A threads; publish to all; amendment notices; clarification log export |
| Evaluation Workspace | Evaluator assignment; scoring interface; blind scoring; matrix view; lock; calibration |
| Decision Workspace | Board configuration; COI declarations; report access; decision recording; approval; immutable record |

**Exit criterion:** A procurement manager can run a complete evaluation and produce an approved, auditable procurement decision.

### 7.4 Phase 4 — Reporting, Administration, and AI

**Goal:** Production-grade platform with full reporting, administration, and embedded AI.

| Module | Key Deliverables |
|---|---|
| Reports | All report types; version-stamped exports; audit trail |
| Administration | User management; roles; configuration; audit log viewer |
| AI Assistant | Quality scores; AI panels in all workspaces; answer drafting; calibration alerts |

**Exit criterion:** The platform is production-ready for enterprise deployment with full compliance documentation, user management, and embedded AI assistance.
